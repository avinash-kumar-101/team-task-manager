PRD — Section 1

Product Overview

Team Task Manager is a collaborative, full-stack web application that enables teams to create projects, manage tasks, assign work to team members, and track overall progress — all through a clean, role-aware interface.

It targets small-to-medium engineering and product teams that need structured work management without the overhead of enterprise tools.

📌 Problem Statement
Teams working without a centralized task system lose velocity to miscommunication, forgotten work, and unclear ownership.
Existing tools are either too heavyweight (Jira) or lack proper role-based access. This product fills the mid-market gap.

PRD — Section 3

User Personas & Role Definitions

Admin

  • Create, edit, and delete Projects
  • Invite users and assign Member role
  • Create, assign, edit, delete any Task
  • View all dashboards and analytics
  • Promote Member to Admin or remove users

Member

  • View Projects they belong to
  • Create Tasks within those Projects
  • Update status of Tasks assigned to them
  • Cannot delete Projects or manage team
  • Cannot access Projects they are not part of

Unauthenticated User

  • Can only access the Signup and Login pages
  • All other routes redirect to Login

Constraints & Assumptions

7.1 Constraints

  • Must deploy on Railway — no other cloud provider accepted for submission
  • No third-party auth providers (no Google OAuth) — custom JWT only
  • Single codebase — monorepo preferred for Railway deployment simplicity

7.2 Assumptions

  • Users have a valid email address for registration
  • A user can belong to multiple projects (v1 scopes role per workspace)
  • No real-time websocket updates in v1 — polling or manual refresh is acceptable

TRD — Section T4

Authentication & RBAC

T4.1 JWT Flow

  • On successful login → server signs JWT with { userId, email, role, exp: 24h }
  • Client stores token (localStorage or memory)
  • Every API call sends token in Authorization: Bearer header
  • auth.js middleware verifies token, attaches req.user — rejects with 401 on failure

T4.2 RBAC Middleware Pattern

// rbac.js — requireRole middleware const requireRole = (...roles) => (req, res, next) => { const memberRole = req.projectMember?.role; if (!roles.includes(memberRole)) { return res.status(403).json({ error: 'Forbidden' }); } next(); }; // Usage in route: router.delete( '/:id', auth, ← verify JWT loadProjectMember, ← attach membership + role requireRole('admin'), deleteProject );

T4.3 loadProjectMember Middleware

For any route with :projectId or :taskId, this middleware queries project_members to confirm the user belongs to the project and fetches their role. This powers all per-project RBAC decisions.

T7.2 State Management

  • AuthContext + useReducer — global auth state (user, token, login, logout)

  • useState — local component state for forms and UI

T7.3 API Service Layer

All HTTP calls centralized in /frontend/src/api/. Axios instance with interceptors: request interceptor attaches JWT, response interceptor handles 401 → auto-logout.
